Introduction

This document will describe how the MAS 500 builder process works and what features are available.  If you have any questions, please contact one of the following:

  • Wei Li

Background

Today we have well over 1000 Visual Basic, Visual C++, and Managed projects that make up the MAS 500 product suite.  A VB project may include source code that is shared by many other projects.  For example; the general.bas module is shared by almost every VB project in MAS 500 today.  In addition, each VB project contains references to an array of ActiveX controls and OLE Automation servers (COM Objects).  Finally, when a VB project is compiled there are many parameters that must be set consistently across the product like Version numbers, Version information, etc.

When VB projects are complied on many different machines it is very easy to have inconsistencies.  For example; if the machines do not have the same common source, controls, or the developers do not set the compilation options correctly, the VB project will be compiled with the incorrect information.

It is also important to make sure we compile every project that must go into the final product.  If we miss a project or compile an old one we reduce the quality of the product.

Does Source Safe contain the most current tested source?  The answer is no.  While Source Safe contains the most current source in general, the source is not always tested.  Many developers use Source Safe as a backup or to manage their changes.

The Solution

To address these problems the build process will incorporate the following features and procedures.

  1. The build process will run unattended on a daily basis.
  2. The build process will by default perform incremental builds.  This means that VB projects will not be extracted from Source Safe and compiled unless identified by the developer, project manager, or configuration administrator.
  3. The build process will build every VB project when instructed to or when shared components are changed.  This includes common source and object references that have new class id's.
  4. Developers and project managers will explicitly promote their projects into the build process.

The process

The following are the steps to the process.  Please review the steps and complete them as instructed.

For all users

  1. Locate the Builder's main web page.  This can be found from http://tahoe or http://afbuilderxx/builder where xx is the MAS 500 version number.  For example http://afbuidler30/builder.
  2. From the contents page (shown on the right) select the task you would like to perform.
  3. Register with the build process by selecting registration from the Contents.  Enter your name and email name (the one in exchange).  For example; John Grant.  Do not use your SMTP address (i.e. johng@sota.com)
  4. Have your manager set your role.
  5. View the build status regularly.
  6. View the build schedule.

For Developers

  1. Register with the build process.  See For all users.
  2. Promote your code when you have unit tested the code.
  3. View the build status.  Determine what projects failed and fix them.

For Program Managers

  1. Register with the build process.  See For all users.
  2. Maintain your projects, add, delete, or change information about your project.
  3. Assign developers to projects.  Each user must have their role set.
  4. Review the promoted code.  If you need to promote your team's code then do so.
  5. View the build status of the projects your team is responsible for.

The Details

The details will describe each task and how to perform them.

Developer Assignment

Developer assignment is simple.  When you first enter this page all the projects you are responsible for will be displayed.  They will show the developer assigned, the name for the project, and the description.
You can change the developer on any entry to any other developer you wish by selecting them from the combo box under the Developer column and pressing the Update button.  You can also query any other module, project manager, or developer by selecting the value in the combo boxes: Manager, Module, and Developer, then pressing the Query button.  Note: you can only update the developer assignment for projects assigned to you.

Manager Assignment

Manager assignment works similar to Developer Assignment in that all the projects you are currently assigned to will be displayed.
The difference with Manager assignment is that any manager can assume the Project Manager role of any project.  This is designed to allow managers to make the needed decisions during busy times.  As a rule, managers should not take control of a project unless agreed upon by the current project manager.

Roles Assignment

Role assignment is an administrative function and should only be executed by managers and administrators.  Access to this page is controlled by IIS and NTFS security privileges.

Project Promotion

Project promotion is used to explicitly identify to the build process when to extract new source code.  When the build process runs it will extract (get) from Source Safe the latest version of the code for any project that has a "promote" status.  If the project was identified as a project to compile the build process will compile that project after the project is extracted from Source Safe.
To change a project's promotion status, simply change the value of the combo box on the left to the value desired.  Then when all values are set correctly, press the update button.

Note: the "Promote All" and "Demote All" buttons will promote or demote all projects on the page.  This will change the current status of each project and there is no undo so be careful.

The project promotion page works differently depending on your role.
Developers If your role is "Developer", when the Project Promotion page is displayed, only projects assigned to you will be displayed.  You can change the query set to filter the projects by module.
Project Manager If your role is "Project Manager", when the Project Promotion page is displayed no items will be displayed until you press the query button.  Project managers can promote any project.  It is expected that developers will promote their own project, but in the absence of the developer the project manager can promote the project.
Administrator If your role is "Administrator" you can promote any project.

Project Maintenance

Project Maintenance allows a Project Manager or Administrator to add new projects to the Builder database.  The projects can be of various types and each type has specific settings.  The project entry describes to the Builder one or more of the following pieces of information:
  • What directory or file to extract from Source Safe and should the extraction be recursive (include subdirectories).
  • What Source Safe database to use.
  • What order to get the project file and if the source is shared by other projects.
  • What projects to compile.
  • If the project is a shared runtime library (Shared dll or ocx).
  • Where to put the extracted file or compiled output.
Use the find button to locate other projects.

Projects must be set to Obsolete before they can be deleted.  Obsolete projects are not included in the build process.
